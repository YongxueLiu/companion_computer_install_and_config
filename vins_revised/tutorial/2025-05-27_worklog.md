# 2025-05-27 工作记录

## 目标
将 VINS-Fusion-ROS2 从 ROS2 Foxy 适配到 **Ubuntu 22.04 + ROS2 Rolling**，并完成 Intel RealSense D435i 的跑通验证。

---

## 一、ROS2 Rolling API 适配

### 1.1 `rclcpp::Node::make_shared` 被移除
Rolling 中删除了该工厂函数，改为标准 `std::make_shared`。

**修改文件：**
- `vins/src/rosNodeTest.cpp`
- `loop_fusion/src/pose_graph_node.cpp`
- `global_fusion/src/globalOptNode.cpp`

### 1.2 `loop_fusion` 参数解析修复
原代码直接读取 `argv[1]`，在 ROS2 下不可靠。统一改为 `rclcpp::remove_ros_arguments(argc, argv)` 方式提取配置文件路径。

### 1.3 `ament_target_dependencies` 被完全移除
Rolling 中不再提供该宏。自研兼容函数 `ament_target_dependencies_compat`，同时兼容：
- 传统 `_INCLUDE_DIRS` / `_LIBRARIES` 包（rclcpp、std_msgs 等）
- 仅导出 CMake targets 的包（cv_bridge、image_transport、ament_index_cpp）

**涉及 4 个 `CMakeLists.txt`**（camera_models、vins、loop_fusion、global_fusion）。

### 1.4 C++ 标准升级
所有包的 `CMAKE_CXX_STANDARD 14 → 17`。

## 三、TF 发布修复

`visualization.cpp` 中的 `pubTF` 被写死 `return; // tmp.`，导致 VINS 不发布 TF，RViz 报 `Frame [world] does not exist`。

修复内容：
1. 删除 `return; // tmp.`，恢复 TF 发布逻辑
2. 在 `registerPub` 中正确初始化 `tf2_ros::TransformBroadcaster`（原代码只声明了空指针，调用 `sendTransform` 时 segfault）
3. 恢复了 `world → body → camera` 的完整 TF 链

## 四、话题名统一

VINS 原话题名为相对路径（如 `point_cloud`），在 Rolling 下与 RViz 默认配置的 `/vins_estimator/point_cloud` 不匹配。

**修改：** 所有 publisher 的话题名前强制加上 `/vins_estimator/` 前缀，包括：

## 五、RealSense D435i 配置调优

### 5.1 话题名匹配
RealSense ROS2 v4.58.1 实际发布的话题为 `/camera/camera/infra1/image_rect_raw`（双重前缀），原配置 `/camera/infra1/image_rect_raw` 无法接收。

**修改 `realsense_stereo_imu_config.yaml`：**
```yaml
image0_topic: "/camera/camera/infra1/image_rect_raw"
image1_topic: "/camera/camera/infra2/image_rect_raw"
```

### 5.2 分辨率匹配
相机默认输出 848×480，原配置 640×480 不匹配。

```yaml
image_width: 848
image_height: 480
```

### 5.3 输出路径
原路径 `/home/dji/output/` 不存在，改为：
```yaml
output_path: "/home/lyx/output/"
```

### 5.4 IMU 兼容性问题
`librealsense2 v2.58.1` 在 Ubuntu 22.04 上存在 `Motion Module failure` HID 电源管理冲突。当前 workaround：
- VO 模式（`imu: 0`）可稳定运行
- VIO 模式（`imu: 1`）需降级 librealsense2 到 v2.54.2（待后续处理）
