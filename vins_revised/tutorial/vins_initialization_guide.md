# VINS 初始化数学模型与实践指南

本文档介绍 VINS-Fusion 在**纯双目 VO 模式**（`imu: 0`）下的初始化原理、数学模型和实践经验，帮助理解为什么对着白墙会失败、为什么系统会尺度发散。

---

## 1. 为什么需要初始化？

VINS-Fusion 是一个基于**非线性优化**的视觉惯性里程计系统。在非线性优化开始前，系统需要一组**初始值**：
- 相机位姿（位置和旋转）
- 特征点的 3D 坐标（地图点）

如果没有合理的初始值，BA（Bundle Adjustment）优化器会收敛到错误的解，导致**尺度漂移**或完全跟踪丢失。

### 单目 VO vs 双目 VO 的尺度问题

| 模式 | 尺度来源 | 尺度是否绝对 |
|------|---------|------------|
| **单目 VO** | 两帧之间的相对运动 | ❌ 不确定（尺度模糊） |
| **双目 VO** | 已知基线的双目三角化 | ✅ 绝对尺度 |

纯双目 VO 之所以能恢复**绝对尺度**，是因为左右目之间的物理基线（约 5cm）是已知的。利用这个已知基线，可以通过**双目视差**直接计算出特征点的真实深度。

## 2. `world` 坐标系是什么？

VINS 在初始化成功时会建立一个名为 **`world`** 的坐标系。它**不是地球坐标系**，也不是固定在某个建筑或地图上的绝对坐标系，而是 VINS **以初始化时刻的相机位置为原点**建立的**相对坐标系**。

### 2.1 world 坐标系的定义

```
world 原点 (0,0,0) = 初始化成功时相机所在的位置
world Z 轴           = 朝上（VIO 模式下与重力方向对齐）
world X 轴           = 与初始化时相机的朝向有关
world Y 轴           = 与初始化时相机的朝向有关
```

### 2.2 纯 VO 模式 vs VIO 模式下的区别

| 模式 | world Z 轴方向 | world X/Y 轴方向 |
|------|---------------|-----------------|
| **纯双目 VO** (`imu: 0`) | 大致朝上（取决于第一帧姿态，不严格对齐重力） | 与第一帧相机的水平朝向一致 |
| **VIO** (`imu: 1`) | 严格与重力方向对齐（通过 IMU 加速度计测量） | 与初始化时相机的水平朝向一致 |

### 2.3 为什么初始位姿是 `[0,0,0]` 和 `[0,0,0,1]`？

初始化成功后，第一帧相机的状态被设为：

| 变量 | 含义 | 格式 | 初始值 | 说明 |
|------|------|------|--------|------|
| `Ps[0]` | 位置（平移） | `[x, y, z]` (米) | `[0, 0, 0]` | 相机就在 world 原点 |
| `Rs[0]` | 旋转（姿态） | 旋转矩阵 / 四元数 | `I` (单位阵) | 四元数 `[0, 0, 0, 1]`，表示无旋转 |

> ⚠️ **注意**：不要把四元数和平移向量搞混。`[1, 0, 0, 0]` 是四元数的一种写法（表示绕 X 轴旋转 180°），**不是位置**。VINS 日志中的输出格式是 `t: x y z q: qx qy qz qw`。

### 2.4 以后所有估计的位姿都是相对于 world 的

初始化完成后，VINS 输出的所有位姿都是 **world → body（相机/IMU）** 的变换：

- **位置 `t`**：相机在 world 坐标系中的坐标 `[x, y, z]`（单位：米）
- **姿态 `q`**：相机相对于 world 的旋转四元数 `[qx, qy, qz, qw]`
- **Path**：历史轨迹，每个点都是相机在不同时刻的 world 坐标

```
world (固定不动)
  └── body (相机，随时间移动)
        └── camera (相机坐标系，与 body 有固定外参关系)
```

**关键理解**：world 坐标系一旦建立就**固定不动**。你拿着相机走动时，变化的是 `body` 相对于 `world` 的位姿，而不是 world 本身。

## 3. 纯双目 VO 初始化流程

VINS 在 `imu: 0` 时的初始化逻辑位于 `estimator.cpp`：

```cpp
if(STEREO && !USE_IMU)
{
    f_manager.initFramePoseByPnP(frame_count, Ps, Rs, tic, ric);
    f_manager.triangulate(frame_count, Ps, Rs, tic, ric);
    optimization();

    if(frame_count == WINDOW_SIZE)
    {
        optimization();
        updateLatestStates();
        solver_flag = NON_LINEAR;
        slideWindow();
        ROS_INFO("Initialization finish!");
    }
}
```

流程可以概括为三步：

```
图像输入 → 特征点跟踪 → PnP 求解位姿 → 双目三角化 → BA 优化 → 初始化完成
```

## 4. 数学模型详解

### 3.1 特征点跟踪（Feature Tracking）

VINS 使用 **Shi-Tomasi 角点检测** + **LK 光流跟踪**：

1. 在左目图像中检测角点（`cv::goodFeaturesToTrack`）
2. 用金字塔 LK 光流跟踪到下一帧左目图像
3. 用反向 LK 光流验证跟踪质量（`FLOW_BACK`）
4. 对双目图像，用光流从左目跟踪到右目

**输出**：左右目图像中匹配的特征点像素坐标。

### 3.2 像素坐标归一化

特征点坐标不是直接用像素值，而是先通过相机内参**归一化**到相机坐标系：

```
x = (u - cx) / fx
y = (v - cy) / fy
z = 1
```

这样得到的 $(x, y, 1)$ 是射线方向向量。VINS 的 `liftProjective` 函数完成这个转换。

### 3.3 双目三角化（Triangulation）

这是双目 VO 恢复**绝对尺度**的核心。

**几何原理**：
- 左目和右目同时看到空间中的一个点 P
- P 在左目图像上的投影为 $p_l$，在右目图像上的投影为 $p_r$
- 已知左右目之间的基线 $b$（约 5cm）
- 通过视差 $d = x_l - x_r$ 计算深度：

$$Z = \frac{f_x \cdot b}{d}$$

**代码实现（DLT）**：

VINS 使用 **Direct Linear Transform（DLT）** 进行三角化，而不是简单的视差公式。这是因为相机可能有旋转（虽然 D435i 左右目几乎平行）。

```cpp
design_matrix.row(0) = point0[0] * Pose0.row(2) - Pose0.row(0);
design_matrix.row(1) = point0[1] * Pose0.row(2) - Pose0.row(1);
design_matrix.row(2) = point1[0] * Pose1.row(2) - Pose1.row(0);
design_matrix.row(3) = point1[1] * Pose1.row(2) - Pose1.row(1);
```

对 $4 \times 4$ 的设计矩阵做 SVD，取最小奇异值对应的向量，得到齐次坐标下的 3D 点。

**深度有效性检查**：

```cpp
double depth = localPoint.z();
if (depth > 0)
    it_per_id.estimated_depth = depth;
```

只有 $Z > 0$（点在相机前方）的特征点才被保留。

### 3.4 PnP 位姿估计

当新一帧图像到来时，VINS 使用 **Perspective-n-Point（PnP）** 求解当前帧的位姿：

**已知**：上一帧三角化出的 3D 点 + 当前帧的 2D 观测
**求解**：当前帧的旋转 $R$ 和平移 $t$

VINS 使用 OpenCV 的 `cv::solvePnP`，以迭代法（`SOLVEPNP_ITERATIVE`）求解，初始值设为上一帧的位姿。

```cpp
cv::solvePnP(pts3D, pts2D, K, D, rvec, t, 1);  // 1 = SOLVEPNP_ITERATIVE
```

### 3.5 Bundle Adjustment（BA）

BA 是 VINS 的**后端优化核心**。它同时优化：
- 滑动窗口内所有帧的位姿（$R, t$）
- 所有特征点的 3D 坐标

**目标函数**（最小化重投影误差）：

$$\min_{R,t,X} \sum_{i,j} \rho \left( \| \pi(R_i X_j + t_i) - x_{ij} \|^2 \right)$$

其中：
- $R_i, t_i$：第 $i$ 帧的位姿
- $X_j$：第 $j$ 个特征点的 3D 坐标
- $x_{ij}$：第 $j$ 个点在第 $i$ 帧的观测像素坐标
- $\pi(\cdot)$：投影函数（3D → 2D）
- $\rho(\cdot)$：Huber 鲁棒核函数（抑制外点）

VINS 使用 **Ceres Solver** 进行优化，`solver costs` 就是每次 BA 的耗时。

## 5. 初始化的实践要点

### 4.1 场景纹理决定成败

**核心事实**：纯视觉 SLAM **完全依赖图像纹理**。没有纹理 = 没有特征点 = 无法初始化。

| 场景 | 特征点数量 | 结果 |
|------|-----------|------|
| 白墙/天花板 | 0 ~ 10 | ❌ 无法三角化，尺度发散 |
| 光滑桌面 | 10 ~ 30 | ⚠️ 不稳定，容易漂移 |
| 键盘/书架/砖墙 | 50 ~ 150 | ✅ 稳定运行 |
| 户外建筑/树木 | 100+ | ✅ 非常稳定 |

**为什么白墙会失败？**

白墙的灰度值几乎不变，梯度 $\nabla I \approx 0$。Shi-Tomasi 角点检测的本质是找**图像梯度矩阵的特征值大的位置**：

$$M = \sum_{x,y} w(x,y) \begin{bmatrix} I_x^2 & I_x I_y \\ I_x I_y & I_y^2 \end{bmatrix}$$

在白墙上 $I_x \approx 0, I_y \approx 0$，所以 $M$ 的两个特征值都接近 0，**没有任何角点被检测出来**。

### 4.2 特征点数量的健康指标

终端日志中的 `n_pts size` 是每次优化时参与的特征点数量：

| `n_pts size` | 状态 | 建议 |
|-------------|------|------|
| 100 ~ 150 | ✅ 优秀 | 系统非常稳定 |
| 50 ~ 100 | ✅ 良好 | 正常运行 |
| 30 ~ 50 | ⚠️ 临界 | 避免快速运动 |
| 10 ~ 30 | ❌ 危险 | 立刻换场景 |
| 0 ~ 10 | ❌ 崩溃 | 系统即将发散 |

### 4.3 判断初始化是否成功

**成功标志**：
1. 终端出现 `Initialization finish!`
2. `solver_flag` 变为 `NON_LINEAR`
3. 位姿 `t` 值在厘米级缓慢变化（不是几米几米跳）
4. `solver costs` 稳定在 10~30ms
5. `n_pts size` 稳定在 50+

**失败标志**：
1. `t` 值瞬间跳到几十米
2. `n_pts size` 从 50 骤降到 0
3. 大量 `throw img0/img1`
4. `solver costs` 突然变大（> 100ms）

### 4.4 相机运动方式

**最佳运动**：**缓慢平移**（左/右/前/后）

**避免的运动**：
- 快速旋转：特征点跑出视野，跟踪丢失
- 静止不动：没有足够的视差，三角化精度低
- 纯旋转：双目视差为 0，无法恢复深度

**为什么需要平移？**

三角化需要**视差（parallax）**。相机平移时，近处物体在左右目/前后帧之间的像素位移大，远处物体位移小。这种差异让系统能区分远近，恢复深度。

如果相机**纯旋转**，所有物体（无论远近）的像素位移相同，系统无法判断深度，导致尺度崩溃。

## 6. 常见失败模式与原因

### 5.1 尺度发散（从几厘米变成十几米）

**原因**：特征点太少或分布不均，BA 优化试图拟合错误的数据，导致尺度爆炸。

**典型场景**：对着白墙、灯管、光滑桌面。

**数学解释**：当特征点数量 $n < 6$ 时，PnP 问题的约束不足，求解的位姿自由度没有被充分约束。优化器为了最小化重投影误差，会通过**放大场景尺度**来"解释"观测数据。

### 5.2 大量 `throw img0/img1`

**原因**：左右目图像在 `sync_process` 中时间戳不匹配，缓冲区管理逻辑丢弃了不同步的帧。

**直接原因**：
- RealSense 驱动偶尔丢帧
- VINS 处理速度跟不上图像发布速度，缓冲区堆积
- 左右目图像到达时间不一致

**影响**：大量丢帧导致双目信息不完整，系统退化为单目 VO，尺度不确定。

### 5.3 `solver costs` 突然变大

**原因**：
- 特征点数量突然增加（场景变化大）
- Ceres 优化陷入病态（ill-conditioned），Hessian 矩阵条件数差
- 外点（outliers）太多，Huber 核频繁触发

### 5.4 位姿"卡住"（连续多帧不变）

**原因**：
- 特征点全部丢失，PnP 求解失败
- 系统进入 `failureDetection()`，但 `clearState()` 后没有正确重启
- 滑动窗口内的帧没有足够视差


- ✅ 正常：左右图都布满红/绿点，均匀分布

## 8. 总结

VINS 纯双目 VO 的初始化本质上是：
1. **检测足够的特征点**（Shi-Tomasi + LK 光流）
2. **用双目基线三角化出深度**（DLT）
3. **用 PnP 求解位姿**
4. **用 BA 优化整个系统**

**这四个步骤的每一步都需要足够的特征点支持**。一旦特征点不足（<30），整个链条断裂，尺度必然发散。

**实践中的黄金法则**：
> **相机对着纹理丰富的场景 + 缓慢平移 + n_pts > 50 = 稳定运行**
